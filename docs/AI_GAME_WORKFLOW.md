# THE FOG REMEMBERS AI 游戏生产工作流

本工作流借鉴 godogen 的“规划 -> 实现 -> 运行 -> 截图/证据 -> 修复”循环，但保留当前项目的技术路线：`Next.js + React + TypeScript + HTML5 Canvas + Web Audio`。短期目标不是迁移到 Godot、Bevy 或 Babylon.js，而是用更严格的生产流程把现有原型打磨成可验收的垂直切片。

## 适用范围

- 玩法逻辑：`game/`
- 游戏挂载、HUD、触控和叠层 UI：`components/GameCanvas.tsx`
- 标题页、开始/继续流程：`app/page.tsx`
- 视觉资源：`public/assets/`
- 音频资源：`public/audio/`
- 世界观、资产、QA 和生产文档：`docs/`

## 每轮改进循环

### 1. 任务定义

每个任务开始前先写清楚四件事：

- 玩家可感知的目标：例如“聆听者靠近时必须有明确声音反馈”。
- 涉及文件范围：例如 `game/monster.ts`、`game/audio.ts`、`components/GameCanvas.tsx`。
- 验收方式：手测路径、自动脚本、截图或视频。
- 不做事项：例如“不迁移引擎”“不引入大型状态库”。

### 2. IP 与玩法检查

实现前先对照 `docs/IP_BIBLE.md`：

- 不使用外部恐怖 IP 的正式名称、地名、怪物名或标志物。
- 新怪物必须有行为规则、象征含义、声音或视觉提示。
- 新关卡必须有独立机制，不只换色和增加怪物密度。
- 新文本必须服务于“雾、罪责、声音线索、下沉、记忆”的原创方向。

### 3. 小步实现

优先做小范围、可验证的改动：

- 数据先拆到 `game/data/`，再扩内容。
- UI 改动先保证桌面和移动端都不遮挡核心操作。
- 音频改动先定义总线和触发条件，再替换具体素材。
- 资产接入必须记录来源、用途和授权状态。

### 4. 本地运行

每轮改动后运行：

```bash
npm run build
npm run qa:smoke
```

`qa:smoke` 会启动本地 Next 服务，检查首页是否可访问、标题内容是否存在、关键图片和音频资源是否能读取。它不是完整浏览器测试，但可以快速拦住路由、资源路径和构建层面的低级错误。

### 5. 视觉验收

涉及 UI、Canvas、标题页、移动端或资源时，执行 `docs/QA_CHECKLIST.md` 中的手测路径，并保留至少两张截图：

- 桌面：`1440x900`
- 移动端：`390x844`

截图检查重点：

- Canvas 不是空白。
- HUD 不遮挡主角和关键交互。
- 移动端触控区域不会挡住暂停、返回或主要信息。
- 标题页第一屏清楚显示原创游戏名。

### 6. 复盘记录

任务结束后在提交说明、PR 描述或工作日志中记录：

- 改了什么。
- 怎么验证。
- 还有什么风险。
- 下一轮最应该做什么。

## 推荐任务队列

### P0：稳定垂直切片

- 修复 `README.md` 编码损坏内容。
- 新增 `docs/QA_CHECKLIST.md` 并按清单跑一轮。
- 将章节、线索和仪式文本从 `game/engine.ts` 拆到 `game/data/chapters.ts`。
- 明确前 4 层的终局条件，让玩家能完成一次短流程。

### P1：可感知反馈

- 给声呐扫描、怪物逼近、道具冷却增加更清楚的视觉/音频反馈。
- 增强“聆听者”规则，让玩家能理解静止、噪声和追踪之间的关系。
- 加入关键交互音效：拾取、完成仪式、出口显影、护盾倒计时。

### P2：自动化验收

- 引入 Playwright。
- 增加标题页和游戏页 smoke test。
- 自动截取桌面和移动端截图。
- 检查 Canvas 非空、HUD 可见、按钮可点击。

### P3：资产管线

- 建立 `public/assets/monsters/`、`public/assets/tiles/`、`public/assets/characters/`。
- 给每个资源维护来源、用途、授权和替换计划。
- 先做 4 套关卡调色和纹理，不急着做完整 tileset。
- 外部审图供应商只通过 `.env.local` 配置，底层读取 `FOG_IMAGE_REVIEW_*`，不得把密钥写进代码、文档或提交记录。

## 采用原则

- 保留当前 Canvas 引擎，直到玩法闭环和 QA 稳定。
- 只借鉴 godogen 的流程，不直接套用它的引擎输出。
- 每次改动都必须能被玩家看见、听见或在 QA 中被验证。
- 自动化脚本先覆盖高频错误，再逐步扩到浏览器截图和交互测试。
